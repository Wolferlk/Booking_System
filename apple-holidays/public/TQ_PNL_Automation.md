# TQ & PNL Automated Processing System

## Overview

The system automatically reads emails from three mailboxes, creates bookings from Travel Quotations, and merges Profit & Loss data — with zero manual entry required.

---

## Mailboxes

| Mailbox | Address | Method | Purpose |
|---------|---------|--------|---------|
| Travel Quotation (TQ) | `confirm.booking@aahaas.com` | Microsoft Graph API (webhook / cron 5min) | Receives tour booking quotations from agents |
| P&L — Payable | `accounts.payable@aahaas.com` | IMAP Poll every 30s | Receives PNL spreadsheets with cost data |
| P&L — Receivable | `accounts.receivable@aahaas.com` | IMAP Poll every 30s | Secondary PNL inbox (same processing) |

## How TQ and PNL Are Linked

### Key: Tour Ref ↔ Tour No

| Field | TQ (Travel Quotation) | PNL (Profit & Loss) |
|-------|-----------------------|---------------------|
| Example | `Tour Ref = 469083CNTL` | `Tour No = #469083` |
| Stored as | `bookingRef = 469083CNTL` | Extracted as `469083` |
| Match logic | Strip non-numeric chars from PNL ref | Find booking where `bookingRef` starts with or ends with the numeric part |

**Fallback matching order:**
1. Exact match: `bookingRef = "469083"`
2. Ends-with: `bookingRef` ends with `"469083"` (handles `VN19679` → `19679`)
3. Starts-with: `bookingRef` starts with `"469083"` (handles `469083CNTL` → `469083`)

---

## Order of Arrival — Both Cases Handled

### Case 1: TQ arrives first (normal flow)
```
1. TQ email arrives → booking created instantly (status: GT_REVIEW)
2. PNL email arrives → Tour No matched → cost lines merged into existing booking
3. Ground team reviews complete booking
```

### Case 2: PNL arrives first
```
1. PNL email arrives → no matching TQ found
   → status set to "PNL_WAITING"
   → shown in UI as orange "Awaiting TQ" card
2. TQ email arrives → booking created
3. System auto-retries the waiting PNL → merges immediately
```

## What Each Email Is Processed Into

### Travel Quotation (TQ) Email produces:
- **Booking record** with ref, agent, dates, pax count
- **Passengers** list (adults + children, lead passenger flagged)
- **Flights** (flight number, route, times)
- **Accommodations** (hotel, city, check-in/out, meal plan)
- **Itinerary items** (day-by-day)
- **Emergency contacts**
- **Movement chart** (tour agenda, auto-generated by GPT-4o)

### PNL Email + XLSX produces:
- **PNLLineItem** rows per activity (hotel, transfer, guide, entrance, etc.)
- **Category** assigned by AI: `HOTEL | CRUISE | TRANSPORT | TICKETS | GUIDES | WATER | MEALS | FLIGHT_TICKETS | TAX_FEES | OTHER`
- **Ticket stubs** auto-created for ticketable categories
- **Pax count** updated on the PNL record

## Auto-Processing Behavior

- Mail Inbox page auto-polls all three mailboxes every **30 seconds**
- New unprocessed emails are added to a queue and processed one-by-one (1.5s gap)
- Each email is deduplicated via `processed_email_<graphId>` key in `SystemSetting` table — so the same email is never processed twice even across cron and page visits
- The cron endpoint (`/api/cron/process-mailboxes`) also runs every 5 minutes as a fallback for emails received when the page is not open

## Error Handling

| Scenario | Behavior |
|----------|----------|
| PNL arrives, no TQ exists, no dates in PNL | Returns `PNL_WAITING` — waits for TQ |
| TQ arrives with missing dates | Processing fails, shown as error card |
| XLSX attachment missing | Falls back to GPT-4o body extraction |
| OpenAI fails | PNL categories kept from XLSX; agenda generation falls back to skeleton |
| IMAP connection failure | Returns empty list — page shows last successful fetch |
| Duplicate email received | Skipped via dedup key — no duplicate booking created |
